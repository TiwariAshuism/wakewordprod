## 1. DSP
**New:** Delay estimation for multi-mic arrays (needed for beamforming, already scoped as deferred past Phase 2) has a standard technique not previously named: **GCC-PHAT** (Generalized Cross-Correlation with Phase Transform) — the standard method for time-difference-of-arrival estimation between microphone pairs, robust to reverberation compared to plain cross-correlation. Relevant only when far-field/beamforming work resumes (Phase 5 per `aura_addendum_v4.md`); no action needed now. **General Knowledge.**

## 2. Machine Learning
**New:** The Cioflan et al. on-device domain-adaptation approach and the "few-shot custom wake word" R&D track (both already scoped) both fall under the broader umbrella of **parameter-efficient fine-tuning (PEFT)** — specifically, **LoRA-style low-rank adapters** are a concrete, unexplored mechanism for the on-device adaptation update itself: instead of updating full model weights on-device (memory-expensive), inserting small trainable adapter matrices and updating only those could plausibly fit the <10KB memory envelope more predictably than full fine-tuning. This is genuinely new relative to all four prior documents, which discussed domain adaptation and distillation but never named PEFT/LoRA as the mechanism. Also relevant: **catastrophic forgetting** is the specific named failure mode to test for once on-device continual adaptation is prototyped (does adapting to a new user/room degrade performance on the original training distribution?). **Hypothesis — Needs Experimentation**, add to research backlog.

## 3. TinyML
**New:** Two accelerator targets were never named: **Arm Ethos-U** (Arm's microNPU line, designed specifically to pair with Cortex-M cores for ML workloads at the microcontroller tier — directly relevant to the Cortex-M targets already in scope, as an alternative to pure CMSIS-NN CPU inference) and **Edge TPU** (Google's Coral accelerator — lower priority since it's a discrete/added-hardware component rather than something present on AURA's already-listed target platforms, but worth naming as a "why not" for completeness). **Recommendation:** evaluate Ethos-U specifically for the Cortex-M tier during Phase 2a hardware selection, since it changes the CMSIS-NN-only assumption already baked into the embedded runtime recommendation. **General Knowledge, Needs Experimentation for actual benefit on AURA's model sizes.**

## 4. Production ML / MLOps
**New, not previously named:** **model cards** and **dataset cards** (standardized documentation of a model/dataset's intended use, training data composition, known limitations, and evaluation results) — directly useful given the multi-accent/multilingual fairness concerns already raised in `aura_phase1_audit.md` §6, but the *documentation artifact* itself was never named. **Drift detection** (monitoring whether live production audio characteristics diverge from training-distribution assumptions over time) and **shadow/canary deployment** (running a new model version against live traffic without affecting real triggers, before staged rollout) are concrete refinements to the OTA staged-rollout design in `aura_addendum_v4.md` §6 — canary deployment specifically should precede the staged-percentage rollout already described, not replace it. **Engineering Judgement**, add to production checklist.

## 5. Security
**New, not previously named:** **SBOM (Software Bill of Materials)** and **SLSA (Supply-chain Levels for Software Artifacts)** — increasingly standard/expected for commercial software supply-chain security (relevant given AURA's dependency on numerous third-party components: ONNX Runtime, TFLite Micro, Silero VAD, WebRTC APM, RNNoise, CMSIS libraries — each is a supply-chain dependency that should be tracked). **TrustZone** (Arm's TEE, relevant on Cortex-A-class mobile/embedded chips as the mechanism underlying Android Keystore's hardware backing, already named in `aura_addendum_v4.md` §8 — this fills in *how* Keystore's hardware backing is actually implemented on Arm silicon, which is new mechanistic detail, not a repeat). **Root/jailbreak detection** — relevant if AURA ever treats the device's OS integrity as a precondition for trusting an on-device speaker-verification decision; not previously raised anywhere in the security discussion. **Recommendation:** add SBOM generation to the CI/CD pipeline (§5 of `aura_addendum_v4.md`) as a concrete deliverable; treat TrustZone/root-detection as relevant only if/when speaker verification proceeds past the ADR-005 gate. **General Knowledge.**

## 6. Android
**Entirely new topic area — not previously covered in any of the four documents.** Android's audio stack has multiple APIs with materially different latency characteristics, directly relevant to an always-on audio-capture pipeline: **AAudio** (modern low-latency API, Android 8.0+) and **Oboe** (Google's open-source C++ wrapper around AAudio with automatic fallback to OpenSL ES on older devices) are the current recommended path for low-latency audio capture on Android — General Knowledge, per Google's own Android NDK audio documentation. **OpenSL ES** is the older, lower-level API Oboe falls back to on pre-8.0 devices. Below the app-facing API, **AudioFlinger** (the system audio service) and its **FastMixer** thread path determine whether a given capture/playback path actually achieves low-latency behavior — not all devices/configurations route through the fast path, which is a real, documented source of per-OEM latency variance (**General Knowledge**, well-known Android audio developer pain point, directly relevant to the NNAPI OEM-fragmentation risk already flagged in `aura_investment_committee_report.md` §4). **Recommendation:** use Oboe (not raw AAudio or OpenSL ES directly) as AURA's Android audio-capture layer — this is a genuinely new, concrete engineering decision not present in any prior document; add as a new ADR (see §17 below).

## 7. iOS
**New, narrowly:** **AVAudioEngine** (the modern, recommended Apple audio API, built on **CoreAudio**/**AudioUnits** underneath) is the equivalent Android-Oboe-style decision point for iOS audio capture — previously the iOS section of the prior documents only discussed CoreML/inference, never the audio-capture API choice itself. **Recommendation:** use AVAudioEngine as AURA's iOS audio-capture layer, mirroring the Oboe decision above. **General Knowledge.**

## 8. Embedded Systems
**New:** **FreeRTOS** (the standard RTOS for ESP32/Cortex-M targets, already implied by the ESP32-S3/microWakeWord precedent discussed in prior documents but never explicitly named as AURA's own RTOS choice) vs. **Zephyr** (a newer, more portable RTOS gaining traction across more MCU vendors) is a genuine, previously-unraised decision point. **Recommendation:** FreeRTOS as the default (largest existing ecosystem overlap with ESP32/Espressif tooling already assumed elsewhere in this document set), Zephyr evaluated only if AURA's Cortex-M target list expands beyond Espressif/ST silicon. **Watchdog timers** and **brownout detection** are standard embedded reliability mechanisms not previously named — relevant to the always-on, unattended, long-run-stability requirement already emphasized throughout; add as explicit firmware requirements, not optional hardening. **Flash wear** (relevant specifically to the OTA update design in `aura_addendum_v4.md` §6 — repeated OTA writes to flash have a wear-leveling/lifetime consideration on MCU-class flash that the OTA design didn't address) — **new, concrete gap**: the OTA rollback mechanism should account for flash write-cycle budgets on MCU targets specifically, which was not previously considered. **General Knowledge, Engineering Judgement on AURA-specific impact.**

## 12. Legal
**Material new finding.** All four prior documents referenced "GDPR" generically as a privacy consideration but never named **India's Digital Personal Data Protection Act, 2023 (DPDP Act)** or its implementing **DPDP Rules, 2025** — a significant omission given the team's India base. **Verified, current as of this review:** the DPDP Rules 2025 were notified by India's Ministry of Electronics and Information Technology on **13 November 2025**, giving full operational effect to the 2023 Act via a phased timeline — administrative provisions effective immediately, consent-manager registration from November 2026, and the substantive compliance obligations (consent notices, security safeguards, breach notification, data-principal rights, children's-data provisions) effective **13 May 2027**. The Act has **extraterritorial reach**: it applies to any entity, including one based outside India, that processes the digital personal data of individuals located in India in connection with offering them goods or services — meaning AURA's applicability to this law is not limited to the fact that the team is India-based; it would also apply based on user location alone. Breach-notification obligations under the Rules have **no materiality threshold** (unlike GDPR/CCPA, which generally require a likelihood-of-harm assessment) — on a strict reading, any personal-data breach must be reported, which is a materially different (stricter, in this respect) compliance posture than the GDPR-centric framing implicit in the prior documents. **Recommendation:** add DPDP Act/Rules compliance explicitly to the Priority Fixes/legal-review scope already established in `aura_investment_committee_report.md` §5, not as a GDPR restatement but as a distinct, currently-phasing-in legal regime with its own timeline (full substantive effect May 2027) that AURA's data-handling design (telemetry, enrollment data, voice embeddings) should be built toward now rather than retrofitted later. Also add, more briefly: **CCPA** (California) was never named either, despite being a comparably standard reference point alongside GDPR for any product with US users — add to the same legal-review scope. **Verified** (DPDP timeline/mechanics, via MeitY notification and multiple independent legal-tracking sources); **Engineering Judgement** on the specific implications for AURA's architecture.

## 15. Research — 2025/2026 Papers Not Previously Cited
- **HEiMDaL** (Kundu et al., Apple, ICASSP 2023/arXiv:2210.15425) — proposes an alignment-based classification loss plus an offset loss for jointly detecting *and localizing* (start-time) a wake word in streaming conditions, avoiding the loss-metric mismatch inherent in DNN-HMM hybrid approaches; reports a substantial detection-metric improvement at equivalent memory footprint to existing DNN-HMM-style models, per the authors' own claims. **Relevance to AURA:** directly applicable to the first-stage/cascade design already scoped — wake-word *localization* (not just detection) is needed if AURA's roadmap ever requires precise start-of-utterance timing (e.g., for the "wake word + command" extension flagged as a possible direction in `aura_phase1_audit.md` §5). **Verdict: add to research backlog as a candidate loss-function design for the first-stage model, not yet prototyped.** Confidence: General Knowledge (paper exists, authors' own reported improvement figure — not independently re-verified numerically by this review).
- **GE2E-KWS** (Zhu et al., Google, SLT 2024) — extends the GE2E (Generalized End-to-End) loss, previously cited in `aura_phase1_audit.md` §5 only in the speaker-verification context, to **zero-shot keyword spotting training and evaluation** directly — i.e., applying GE2E-style metric learning to the KWS task itself rather than only to speaker embeddings. **Relevance:** a concrete, recent (2024) precedent for a metric-learning-based (rather than pure classification) approach to the custom/few-shot wake-word problem, complementing (not replacing) the Cioflan-et-al. domain-adaptation and WavLM-distillation approaches already on the research backlog. **Verdict: add as a third candidate approach to the few-shot custom-wake-word research backlog.** Confidence: General Knowledge.
- **LLM-Synth4KWS** (2025, arXiv:2505.22995) — uses large language models to automatically generate confusable-word/phrase candidates at scale for hard-negative training data construction, rather than relying solely on ASR-decoded real audio corpora (the hard-negative mining method already established in `aura_phase1_audit.md` §3.7/§8). **Relevance:** a concrete, recent, complementary technique to the existing hard-negative-mining recommendation — using an LLM to generate the *candidate confusable phrase list* is cheaper than mining it purely from decoded audio, though the resulting phrases still need to be synthesized/recorded as actual audio via the TTS pipeline already scoped. **Verdict: add to the data-pipeline research backlog as a complement to, not a replacement for, ASR-based hard-negative mining.** Confidence: General Knowledge.
- **DMA-KWS** (2026, arXiv:2605.22120) — a dual-stage matching framework for user-defined KWS (CTC-based phoneme search followed by fine-grained phoneme-matcher verification), plus multi-modal (audio+text) enrollment and a lightweight continual-adaptation mechanism. **Relevance:** directly relevant to the custom-wake-word-enrollment problem; the multi-modal (text+audio) enrollment idea specifically — letting a user type the wake phrase as text in addition to recording it, to improve phoneme-level disambiguation — was not previously considered anywhere in this document set and is a concrete, low-cost UX/engineering idea worth prototyping regardless of which underlying model architecture is chosen. **Verdict: add "text+audio multi-modal enrollment" as a specific, novel product/engineering idea to the Phase 3 personalization backlog.** Confidence: General Knowledge (2026 preprint, not yet peer-reviewed at time of this review — flagged as such).
- **EfficientWord-Net** (Chidhambararajan et al., 2022, open-source) — a real, open-source few-shot hotword-detection engine already in existence, not previously named. **Relevance:** worth including directly in the open-source ecosystem competitive audit alongside openWakeWord/microWakeWord/Porcupine (already covered in `aura_research_report.md` §2), specifically because it targets the few-shot custom-wake-word use case AURA is treating as a high-risk R&D item — an existing open-source attempt at this exact problem is directly relevant prior art to study before investing further R&D effort. **Verdict: add to the competitive/open-source audit as a specific few-shot-KWS prior-art reference.** Confidence: General Knowledge.
